[x] T01: CPUID HYPERVISOR BIT
--------------------------------------------------------------------------------
* HOW IT WORKS: The malware executes the `__cpuid(1)` hardware instruction and 
  checks if bit 31 of the ECX register is set to 1. If it is, the CPU is 
  announcing that a hypervisor is present.
* COUNTERMEASURE: Intercept KVM initialization from the Linux Host.
* WHAT YOU DID: Engineered a custom `kvm_hook.so` shared library and injected it 
  into the QEMU process using an `/opt/qemu-wrapper.sh` script (bypassing 
  AppArmor). The hook catches the `KVM_SET_CPUID2` ioctl and explicitly clears 
  bit 31 before the VM boots.
* STATUS: COMPLETED

[ ] T02: MAC ADDRESS VALIDATION
--------------------------------------------------------------------------------
* HOW IT WORKS: The malware calls the Windows `GetAdaptersInfo` API to read the 
  MAC address of the network card. It checks the first 3 bytes (the OUI) against 
  known virtual vendors (e.g., 52:54:00 for QEMU/KVM).
* COUNTERMEASURE: Spoof the virtual network adapter's hardware address.
* WHAT TO DO: Modify the libvirt XML `<interface>` tag to assign a physical 
  hardware vendor's MAC address (e.g., an Intel OUI like 00:1B:21:xx:xx:xx).
* STATUS: PENDING

[ ] T03: DRIVER FILES (VIRTIO/QEMU)
--------------------------------------------------------------------------------
* HOW IT WORKS: The malware uses `GetFileAttributes` to scan the 
  C:\Windows\System32\drivers\ directory for telltale virtual drivers like 
  `viostar.sys`, `vmmouse.sys`, or `balloon.sys`.
* COUNTERMEASURE: Break the file path while keeping the driver functional.
* WHAT TO DO: Inside the Windows guest, manually rename the paravirtualized 
  .sys files to benign names (e.g., `stornvme_ext.sys`) and update their 
  corresponding `ImagePath` keys in the Windows Registry.
* STATUS: PENDING

[ ] T04: ANALYSIS PROCESS SCAN
--------------------------------------------------------------------------------
* HOW IT WORKS: The malware takes a snapshot of running processes using 
  `CreateToolhelp32Snapshot` and loops through them looking for analysis tools 
  (e.g., `wireshark.exe`, `procmon.exe`, `qemu-ga.exe`).
* COUNTERMEASURE: Hide or rename the analysis executables.
* WHAT TO DO: Uninstall the QEMU Guest Agent before detonation, and manually 
  rename any analysis tools (e.g., change `wireshark.exe` to `svchost_net.exe`).
* STATUS: PENDING

[ ] T05: PROCESS COUNT CHECK
--------------------------------------------------------------------------------
* HOW IT WORKS: The malware counts the total number of running processes. A clean, 
  freshly booted Windows VM usually has very few processes running. If the count 
  is under 15, it assumes it's an automated sandbox.
* COUNTERMEASURE: Artificially inflate the system state.
* WHAT TO DO: Write a simple `.bat` script to run inside Windows before malware 
  execution that launches 20 hidden instances of benign applications (like 
  `notepad.exe` or `calc.exe`).
* STATUS: PENDING

[ ] T06: RDTSC TIMING ANALYSIS
--------------------------------------------------------------------------------
* HOW IT WORKS: The malware executes `__cpuid` sandwiched between two `__rdtsc` 
  timestamp checks. Because `__cpuid` causes a VM-Exit context switch to the 
  hypervisor, it takes thousands of cycles longer in a VM than on bare metal.
* COUNTERMEASURE: Neutralize the latency calculation.
* WHAT TO DO: Use a hex editor or Python script to perform Binary Patching on 
  the malware executable inside the guest. Locate the `cmp` instruction checking 
  for 750 cycles, and patch the `JA` (Jump Above) to a `JMP` (Unconditional Jump).
* STATUS: PENDING

[ ] T07: AUDIO DEVICE CHECK
--------------------------------------------------------------------------------
* HOW IT WORKS: The malware calls `waveOutGetNumDevs()` to verify the system 
  has physical audio rendering capabilities (many sandboxes are headless/mute).
* COUNTERMEASURE: Attach simulated audio hardware.
* WHAT TO DO: Add a virtual soundcard to the libvirt XML (e.g., `<sound model='ich9'/>`) 
  so Windows loads a standard audio driver.
* STATUS: PENDING

[x] T08: ACTIVE HARDWARE SIGNAL CHECK (REFRESH RATE)
--------------------------------------------------------------------------------
* HOW IT WORKS: The malware bypasses the logical Windows desktop resolution and 
  uses the `QueryDisplayConfig` CCD API to query the Target Mode. It looks for 
  virtual monitors outputting 0Hz refresh rates, missing targets, or missing EDIDs.
* COUNTERMEASURE: Inject physical monitor hardware parameters.
* WHAT YOU DID: Extracted a physical monitor's raw EDID binary file, saved it 
  to `/opt/real_monitor_edid.bin`, ensured QEMU had read permissions, and 
  injected it into the virtual GPU using the `<edid file='...'/>` XML tag 
  (alongside clamping the resolution to fix the WDDM driver panic).
* STATUS: COMPLETED (Testing Validation in Progress)
