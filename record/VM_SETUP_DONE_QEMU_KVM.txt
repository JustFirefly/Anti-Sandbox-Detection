1.0 VIRTUALBOX MIGRATION DIAGNOSTICS & RESOLUTION
-------------------------------------------------
Objective: Troubleshoot VERR_DISK_FULL error during Windows 11 VM migration to an external drive.

    1.1 Storage & File System Analysis
    - Diagnosed the move failure caused by the external exFAT file system inflating VirtualBox sparse files (expanding 22GB of logical data to physical limits).
    - Located and recovered the split base virtual hard drive (`Target_VM-W11.vdi`) and associated snapshot differential disks left scattered after the migration crash.

    1.2 VM Consolidation
    - Restored the VM registry state by safely unregistering and re-adding the `.vbox` configuration.
    - Performed a Full Clone of the current machine state to merge the base disk and complex snapshot hierarchy into a single, cohesive `.vdi` file.

2.0 QEMU/KVM HYPERVISOR TRANSITION
----------------------------------
Objective: Migrate the consolidated Windows 11 VM from VirtualBox to QEMU/KVM for improved performance and better external drive compatibility.

    2.1 Disk Image Conversion
    - Installed `qemu-utils` on the Ubuntu host.
    - Executed format conversion from VirtualBox Native (`.vdi`) to QEMU Copy-On-Write (`.qcow2`) via the `qemu-img convert` command.
    - Provisioned the resulting `.qcow2` file in the `/media/firefly/VMs/` external directory.

    2.2 Virt-Manager & Windows 11 Provisioning
    - Installed necessary Windows 11 prerequisites on the Ubuntu host, including `swtpm` (Software TPM emulator) and `ovmf` (UEFI firmware).
    - Configured the new Virt-Manager domain with UEFI Secure Boot (`OVMF_CODE_4M.secboot.fd`) and emulated vTPM 2.0 hardware to satisfy Windows 11 boot requirements.

    2.3 Host Security & Permissions Adjustment
    - Resolved `Permission denied` errors by editing `/etc/libvirt/qemu.conf`.
    - Configured the QEMU service to execute under the host user account (`firefly`) to bypass strict libvirt restrictions on mounted media directories.
    - Restarted the `libvirtd` service to apply the new execution context.

3.0 VIRTIO-FS SHARED DIRECTORY IMPLEMENTATION
---------------------------------------------
Objective: Establish high-speed, read/write file sharing between the Ubuntu host and Windows 11 guest utilizing VirtioFS.

    3.1 Host-Side Hardware Configuration
    - Enabled shared memory allocation within the VM hardware configuration to support high-speed proxying.
    - Deployed a VirtioFS filesystem device pointing to the host directory, explicitly ensuring read-only mode was disabled to allow bidirectional transfers.
    - Resolved `Unable to find a satisfying virtiofsd` error by installing the `virtiofsd` daemon on the Ubuntu host.

    3.2 Guest-Side Driver & Service Deployment
    - Removed legacy Oracle VM VirtualBox Guest Additions from the Windows guest to prevent driver conflicts.
    - Mounted the `virtio-win-0.1.285.iso` and installed the core Red Hat QEMU guest tools.
    - Manually deployed the Windows File System Proxy (WinFSP) to provide translation capabilities for the Virtio-FS Service.
    - Force-updated the `viosfs` Mass Storage Controller driver via Windows Device Manager to ensure proper hardware recognition.

    3.3 Final Network Mapping & Service Verification
    - Verified the Virtio-FS Service auto-startup status within the Windows `services.msc` snap-in.
    - Troubleshot `System error 67` (Network name not found) by identifying a target path naming mismatch (`vbox_sShare` vs `vbox_share`).
    - Executed a full VM power-cycle (bypassing Windows Fast Startup hibernation) to flush the old hardware tags.
    - Successfully mapped the shared folder to the `Z:` drive using the command: `net use Z: \\virtiofs\vbox_share /persistent:yes`.
